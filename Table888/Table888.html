Hi, Welcome to the Table888 processor site.

Table888 is a work-in-progress.

Table888 is a 64 bit cpu with 256 general purpose registers. The Table888 book contains more details on the processor.

 

Features:

- 256 general purpose registers

- fixed length 40 bit instructions

- load/store architecture , memory mapped I/O

- two addressing modes - register indirect with displacement and indexed

- multi-stage non-overlapped pipeline

 

Table888 advanced also includes

- memory management

- segmentation with 16 segment registers

- paged memory management, with TLB

- Triple mode redundancy memory access capability (TMR)

 

Software:

64 bit C Compiler - the C compiler hasn't been thoroughly tested yet. It generates what looks like reasonable code.

    The compiler is coming along nicely. It's been able to compile about 5,000 lines of code without problems.

64 bit assembler - this tool was used to build the boot rom.

A rudimentary DOS is the works. It will contain only bare necessities to be able to select a program from SD Card to run.

 

Status:

- The processor has been modified to support triple mode redundant (TMR) memory access as an option. I was having a lot of mysterious problems due to I think my board being partially fried by a lightning strike (two of the switches no longer work so maybe an I/O buffer or two is shot). So I tried adding TMR and it seems to help.

- the processor is running a bootrom in an FPGA

- The system can now boot from SD card.

- the segmentation problem has been found and fixed, traced to a reg not being properly initialized. Synthesis assumed it'd be a one and optimized out the segmentation setting logic. Simulation assumed it'd be a zero.

- Paging has passed a first test at paging. The bootrom was mapped into a high memory area, then accessed with memory dump commands to prove it got mapped correctly.

- The instruction cache needs some more work. It seems not to work on occasion. Once again it seems to be working in simulation but not in real life.

 

Goals

Older Status:

But, the perplexing problem is that it seems to work fine in simulation (the exact same software), so it's going to be a tough one to debug.
